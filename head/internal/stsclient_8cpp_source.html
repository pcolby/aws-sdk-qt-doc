168 /**
169  * Returns a set of temporary security credentials (consisting of an access key ID, a secret access key, and a security
170  * token) that you can use to access AWS resources that you might not normally have access to. Typically, you use
171  * <code>AssumeRole</code> for cross-account access or federation. For a comparison of <code>AssumeRole</code> with the
172  * other APIs that produce temporary credentials, see <a
173  * href="http://docs.aws.amazon.com/IAM/latest/UserGuide/id_credentials_temp_request.html">Requesting Temporary Security
174  * Credentials</a> and <a
175  * href="http://docs.aws.amazon.com/IAM/latest/UserGuide/id_credentials_temp_request.html#stsapi_comparison">Comparing the
176  * AWS STS APIs</a> in the <i>IAM User
177  *
178  * Guide</i>>
179  *
180  * <b>Important:</b> You cannot call <code>AssumeRole</code> by using AWS root account credentials; access is denied. You
181  * must use credentials for an IAM user or an IAM role to call <code>AssumeRole</code>.
182  *
183  * </p
184  *
185  * For cross-account access, imagine that you own multiple accounts and need to access resources in each account. You could
186  * create long-term credentials in each account to access those resources. However, managing all those credentials and
187  * remembering which one can access which account can be time consuming. Instead, you can create one set of long-term
188  * credentials in one account and then use temporary security credentials to access all the other accounts by assuming
189  * roles in those accounts. For more information about roles, see <a
190  * href="http://docs.aws.amazon.com/IAM/latest/UserGuide/roles-toplevel.html">IAM Roles (Delegation and Federation)</a> in
191  * the <i>IAM User Guide</i>.
192  *
193  * </p
194  *
195  * For federation, you can, for example, grant single sign-on access to the AWS Management Console. If you already have an
196  * identity and authentication system in your corporate network, you don't have to recreate user identities in AWS in order
197  * to grant those user identities access to AWS. Instead, after a user has been authenticated, you call
198  * <code>AssumeRole</code> (and specify the role with the appropriate permissions) to get temporary security credentials
199  * for that user. With those temporary security credentials, you construct a sign-in URL that users can use to access the
200  * console. For more information, see <a
201  * href="http://docs.aws.amazon.com/IAM/latest/UserGuide/id_credentials_temp.html#sts-introduction">Common Scenarios for
202  * Temporary Credentials</a> in the <i>IAM User
203  *
204  * Guide</i>>
205  *
206  * By default, the temporary security credentials created by <code>AssumeRole</code> last for one hour. However, you can
207  * use the optional <code>DurationSeconds</code> parameter to specify the duration of your session. You can provide a value
208  * from 900 seconds (15 minutes) up to the maximum session duration setting for the role. This setting can have a value
209  * from 1 hour to 12 hours. To learn how to view the maximum value for your role, see <a
210  * href="http://docs.aws.amazon.com/IAM/latest/UserGuide/id_roles_use.html#id_roles_use_view-role-max-session">View the
211  * Maximum Session Duration Setting for a Role</a> in the <i>IAM User Guide</i>. The maximum session duration limit applies
212  * when you use the <code>AssumeRole*</code> API operations or the <code>assume-role*</code> CLI operations but does not
213  * apply when you use those operations to create a console URL. For more information, see <a
214  * href="http://docs.aws.amazon.com/IAM/latest/UserGuide/id_roles_use.html">Using IAM Roles</a> in the <i>IAM User
215  *
216  * Guide</i>>
217  *
218  * The temporary security credentials created by <code>AssumeRole</code> can be used to make API calls to any AWS service
219  * with the following exception: you cannot call the STS service's <code>GetFederationToken</code> or
220  * <code>GetSessionToken</code>
221  *
222  * APIs>
223  *
224  * Optionally, you can pass an IAM access policy to this operation. If you choose not to pass a policy, the temporary
225  * security credentials that are returned by the operation have the permissions that are defined in the access policy of
226  * the role that is being assumed. If you pass a policy to this operation, the temporary security credentials that are
227  * returned by the operation have the permissions that are allowed by both the access policy of the role that is being
228  * assumed, <i> <b>and</b> </i> the policy that you pass. This gives you a way to further restrict the permissions for the
229  * resulting temporary security credentials. You cannot use the passed policy to grant permissions that are in excess of
230  * those allowed by the access policy of the role that is being assumed. For more information, see <a
231  * href="http://docs.aws.amazon.com/IAM/latest/UserGuide/id_credentials_temp_control-access_assumerole.html">Permissions
232  * for AssumeRole, AssumeRoleWithSAML, and AssumeRoleWithWebIdentity</a> in the <i>IAM User
233  *
234  * Guide</i>>
235  *
236  * To assume a role, your AWS account must be trusted by the role. The trust relationship is defined in the role's trust
237  * policy when the role is created. That trust policy states which accounts are allowed to delegate access to this
238  * account's role.
239  *
240  * </p
241  *
242  * The user who wants to access the role must also have permissions delegated from the role's administrator. If the user is
243  * in a different account than the role, then the user's administrator must attach a policy that allows the user to call
244  * AssumeRole on the ARN of the role in the other account. If the user is in the same account as the role, then you can
245  * either attach a policy to the user (identical to the previous different account user), or you can add the user as a
246  * principal directly in the role's trust policy. In this case, the trust policy acts as the only resource-based policy in
247  * IAM, and users in the same account as the role do not need explicit permission to assume the role. For more information
248  * about trust policies and resource-based policies, see <a
249  * href="http://docs.aws.amazon.com/IAM/latest/UserGuide/access_policies.html">IAM Policies</a> in the <i>IAM User
250  *
251  * Guide</i>>
252  *
253  * <b>Using MFA with AssumeRole</b>
254  *
255  * </p
256  *
257  * You can optionally include multi-factor authentication (MFA) information when you call <code>AssumeRole</code>. This is
258  * useful for cross-account scenarios in which you want to make sure that the user who is assuming the role has been
259  * authenticated using an AWS MFA device. In that scenario, the trust policy of the role being assumed includes a condition
260  * that tests for MFA authentication; if the caller does not include valid MFA information, the request to assume the role
261  * is denied. The condition in a trust policy that tests for MFA authentication might look like the following
262  *
263  * example>
264  *
265  * <code>"Condition": {"Bool": {"aws:MultiFactorAuthPresent": true}}</code>
266  *
267  * </p
268  *
269  * For more information, see <a href="http://docs.aws.amazon.com/IAM/latest/UserGuide/MFAProtectedAPI.html">Configuring
270  * MFA-Protected API Access</a> in the <i>IAM User Guide</i>
271  *
272  * guide>
273  *
274  * To use MFA with <code>AssumeRole</code>, you pass values for the <code>SerialNumber</code> and <code>TokenCode</code>
275  * parameters. The <code>SerialNumber</code> value identifies the user's hardware or virtual MFA device. The
276  * <code>TokenCode</code> is the time-based one-time password (TOTP) that the MFA devices produces.
277  *
278  * @param request Request to send to AWS Security Token Service.
279  *
280  * @return A pointer to a related response object.
281  *
282  * @note The caller is to take responsbility for the resulting pointer.
283  */
