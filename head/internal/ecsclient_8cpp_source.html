93 /**
94  * @class EcsClient
95  *
96  * @brief Client for Amazon EC2 Container Service ( ECS)
97  *
98  * Amazon Elastic Container Service (Amazon ECS) is a highly scalable, fast, container management service that makes it
99  * easy to run, stop, and manage Docker containers on a cluster. You can host your cluster on a serverless infrastructure
100  * that is managed by Amazon ECS by launching your services or tasks using the Fargate launch type. For more control, you
101  * can host your tasks on a cluster of Amazon Elastic Compute Cloud (Amazon EC2) instances that you manage by using the EC2
102  * launch type. For more information about launch types, see <a
103  * href="http://docs.aws.amazon.com/AmazonECS/latest/developerguide/launch_types.html">Amazon ECS Launch
104  *
105  * Types</a>>
106  *
107  * Amazon ECS lets you launch and stop container-based applications with simple API calls, allows you to get the state of
108  * your cluster from a centralized service, and gives you access to many familiar Amazon EC2
109  *
110  * features>
111  *
112  * You can use Amazon ECS to schedule the placement of containers across your cluster based on your resource needs,
113  * isolation policies, and availability requirements. Amazon ECS eliminates the need for you to operate your own cluster
114  * management and configuration management systems or worry about scaling your management
115  */

199 /**
200  * Runs and maintains a desired number of tasks from a specified task definition. If the number of tasks running in a
201  * service drops below <code>desiredCount</code>, Amazon ECS spawns another copy of the task in the specified cluster. To
202  * update an existing service, see
203  *
204  * <a>UpdateService</a>>
205  *
206  * In addition to maintaining the desired count of tasks in your service, you can optionally run your service behind a load
207  * balancer. The load balancer distributes traffic across the tasks that are associated with the service. For more
208  * information, see <a
209  * href="http://docs.aws.amazon.com/AmazonECS/latest/developerguide/service-load-balancing.html">Service Load Balancing</a>
210  * in the <i>Amazon Elastic Container Service Developer
211  *
212  * Guide</i>>
213  *
214  * You can optionally specify a deployment configuration for your service. During a deployment, the service scheduler uses
215  * the <code>minimumHealthyPercent</code> and <code>maximumPercent</code> parameters to determine the deployment strategy.
216  * The deployment is triggered by changing the task definition or the desired count of a service with an
217  * <a>UpdateService</a>
218  *
219  * operation>
220  *
221  * The <code>minimumHealthyPercent</code> represents a lower limit on the number of your service's tasks that must remain
222  * in the <code>RUNNING</code> state during a deployment, as a percentage of the <code>desiredCount</code> (rounded up to
223  * the nearest integer). This parameter enables you to deploy without using additional cluster capacity. For example, if
224  * your service has a <code>desiredCount</code> of four tasks and a <code>minimumHealthyPercent</code> of 50%, the
225  * scheduler can stop two existing tasks to free up cluster capacity before starting two new tasks. Tasks for services that
226  * <i>do not</i> use a load balancer are considered healthy if they are in the <code>RUNNING</code> state. Tasks for
227  * services that <i>do</i> use a load balancer are considered healthy if they are in the <code>RUNNING</code> state and the
228  * container instance they are hosted on is reported as healthy by the load balancer. The default value for
229  * <code>minimumHealthyPercent</code> is 50% in the console and 100% for the AWS CLI, the AWS SDKs, and the
230  *
231  * APIs>
232  *
233  * The <code>maximumPercent</code> parameter represents an upper limit on the number of your service's tasks that are
234  * allowed in the <code>RUNNING</code> or <code>PENDING</code> state during a deployment, as a percentage of the
235  * <code>desiredCount</code> (rounded down to the nearest integer). This parameter enables you to define the deployment
236  * batch size. For example, if your service has a <code>desiredCount</code> of four tasks and a <code>maximumPercent</code>
237  * value of 200%, the scheduler can start four new tasks before stopping the four older tasks (provided that the cluster
238  * resources required to do this are available). The default value for <code>maximumPercent</code> is
239  *
240  * 200%>

243  *
244  * logic> <ul> <li>
245  *
246  * Determine which of the container instances in your cluster can support your service's task definition (for example, they
247  * have the required CPU, memory, ports, and container instance
248  *
249  * attributes)> </li> <li>
250  *
251  * By default, the service scheduler attempts to balance tasks across Availability Zones in this manner (although you can
252  * choose a different placement strategy) with the <code>placementStrategy</code>
253  *
254  * parameter)> <ul> <li>
255  *
256  * Sort the valid container instances by the fewest number of running tasks for this service in the same Availability Zone
257  * as the instance. For example, if zone A has one running service task and zones B and C each have zero, valid container
258  * instances in either zone B or C are considered optimal for
259  *
260  * placement> </li> <li>
261  *
262  * Place the new service task on a valid container instance in an optimal Availability Zone (based on the previous steps),
263  * favoring container instances with the fewest number of running tasks for this
264  *
265  * @param request Request to send to Amazon EC2 Container Service.
266  *
267  * @return A pointer to a related response object.
268  *
